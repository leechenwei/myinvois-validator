MyInvois JSON sample & format (UBL 2.1, v1.1)

A clean, correct Malaysia e-invoice JSON you can copy, paste, and adapt — plus the one formatting quirk that trips everyone up. When you're done, validate your version in your browser (nothing uploaded).

The #1 gotcha: every value is wrapped

MyInvois JSON follows the UBL 2.1 structure. Unlike normal JSON, every field value is an array of objects, and the value lives under a "_" key. Attributes (like currencyID or schemeID) sit alongside the "_".

✗ Normal JSON (rejected)

"ID": "INV-001",
"DocumentCurrencyCode": "MYR"

✓ MyInvois UBL JSON

"ID": [{ "_": "INV-001" }],
"DocumentCurrencyCode": [{ "_": "MYR" }]

The document is also wrapped in three namespace keys (_D, _A, _B) and an Invoice array.

Complete minimal sample (Invoice v1.1)

This validates structurally against the v1.1 mandatory fields. Replace the values with your own (and add your digital signature in the signing step before submitting).

    "IssueTime": [{ "_": "09:15:00Z" }],

    "DocumentCurrencyCode": [{ "_": "MYR" }],

Mandatory pieces to remember

FieldRule
IDInvoice number, ≤ 50 chars, unique per supplier
IssueDate / IssueTimeYYYY-MM-DD / HH:MM:SSZ (UTC)
InvoiceTypeCodee.g. 01, with listVersionID "1.1"
DocumentCurrencyCodeISO 4217, e.g. MYR (never RM)
Supplier & Buyer TINschemeID TIN; plus a registration ID (BRN / NRIC / PASSPORT)
Supplier MSIC5-digit IndustryClassificationCode
InvoiceLineItem description + LineExtensionAmount (currencyID = doc currency)
LegalMonetaryTotalTaxExclusiveAmount, TaxInclusiveAmount, PayableAmount
