Why your MyInvois e-Invoice was rejected — and how to fix it

The MyInvois (LHDN) system rejects an e-invoice rather than warn you, often with a terse code. Below are the most common reasons a Malaysian e-invoice JSON fails validation, in plain English, with the exact fix. Paste your file into the free validator to find which one is yours — it runs in your browser, nothing is uploaded.

1. Invalid Structure — fields not wrapped correctly

The #1 mistake when hand-building MyInvois JSON. UBL 2.1 requires every value to be wrapped in an array of objects with a "_" key. A bare value is rejected.

✗ Wrong

"ID": "INV-001"

✓ Correct

"ID": [{ "_": "INV-001" }]

2. "ISO Currency Code ... not updated" — RM instead of MYR

The currency must be the 3-letter ISO 4217 code MYR, not RM. Every monetary amount's currencyID attribute must match the document currency too.

✗ "DocumentCurrencyCode": [{ "_": "RM" }]

✓ "DocumentCurrencyCode": [{ "_": "MYR" }]

3. Foreign currency without an exchange rate

Since 1 September 2025, a non-MYR invoice must include the currency exchange rate to MYR, or it is rejected. For an MYR invoice the rate must be left blank or 0 (a random rate is also rejected). Add the TaxExchangeRate with CalculationRate for foreign-currency invoices.

4. "Buyer TIN is invalid" / TIN is required

Both supplier and buyer need a valid Tax Identification Number (TIN). Common causes: the TIN is missing, has the wrong length, or uses an outdated prefix. Don't guess the TIN — get the correct one from LHDN's official Search TIN function in the MyTax portal. For business-to-consumer consolidated invoices, the general-public buyer TIN is EI00000000010.

5. "Classification code is required" / invalid

Each invoice line needs a mandatory 3-digit classification code (001–045) describing the item/service. It is separate from the supplier's 5-digit MSIC code. Missing or out-of-range codes are rejected.

6. Wrong date or time format

Date must be YYYY-MM-DD and time must be HH:MM:SSZ in UTC (note the trailing Z). Local formats like 20/06/2026 are rejected.

7. DS302 — Duplicated Submission

MyInvois returns DS302 when you submit a document whose key attributes (invoice number + supplier TIN) match one already submitted. Each invoice number must be unique per supplier; never resubmit the same document — if it was already Valid, use the existing one.

8. Missing or invalid buyer email / phone

Recent (2025–2026) validation rules check the buyer's email and phone fields. Provide a valid email and a phone number in international +60… (E.164) format, or NA where genuinely not applicable, per the latest LHDN rules.

9. Consolidated invoice over RM10,000

Since January 2026, a single transaction above RM10,000 cannot be issued as a consolidated / general-public invoice — it must be a normal e-invoice with the buyer's real details (real TIN, name, address), not the EI00000000010 general-public TIN.

About the coded errors (CFxxx, DS3xx, Structure)

MyInvois groups failures into validators: a Structure Validator (UBL 2.1 shape), a Core Fields Validator (the CFxxx codes — a mandatory field is missing or malformed), a Code Validator (invalid code-table values), a Currency Validator, and a Duplicate Submission Validator (DS302). Whatever the code, it points to one specific field failing a rule above — match the field named in the error to the fix here. For the authoritative list, see LHDN's official validation-rules documentation.
